## Getting Started

### 1. Install PHP dependencies

```bash
composer install
```

### 2. Install frontend dependencies

```bash
npm install
```

### 3. Copy the environment file

```bash
cp .env.example .env
```

Fill in your database credentials and set a value for `APP_KEY`.

### 4. Run migrations

```bash
php espresso migrations:migrate
```

### 5. Start the development servers

In one terminal, start the PHP server:

```bash
php espresso serve
```

In another terminal, start the Vite asset server:

```bash
npm run dev
```

Open [http://localhost:8000](http://localhost:8000) in your browser.

## Generator Commands

The `espresso` CLI tool generates boilerplate files for you.

```bash
php espresso <command> <Name>
```

| Command | What it generates |
|---|---|
| `make:controller Post` | `src/Http/Controllers/PostController.php` |
| `make:service Post` | `src/Services/PostService.php` |
| `make:repository Post` | `src/Database/Repository/PostRepository.php` |
| `make:request StorePost` | `src/Http/Requests/StorePostRequest.php` |
| `make:entity Post` | `src/Database/Entities/Post.php` |
| `make:migration CreatePostsTable` | `database/migrations/Version{timestamp}.php` |

**Other CLI commands:**

```bash
php espresso serve                       # Start the development server (localhost:8000)
php espresso serve --host=0.0.0.0        # Bind to all interfaces
php espresso serve --port=8080           # Use a custom port
php espresso migrations:migrate          # Run all pending migrations
php espresso migrations:diff             # Generate migration from entity changes
php espresso migrations:status           # Show migration status
php espresso migrations:generate         # Generate a blank migration
php espresso migrations:execute VERSION  # Run a specific migration version
```

## Request Pipeline

Every HTTP request flows through these layers in order:

```
Route → Controller → FormRequest (validation) → Service → Repository → Database
```

Think of it as a production line:

- **Route** — decides which controller handles the URL
- **Controller** — receives the request, validates input, calls the service
- **FormRequest** — validates and sanitizes incoming data; rejects bad input early
- **Service** — contains all business logic; does not touch the database directly
- **Repository** — the only layer that talks to the database

Keeping each layer focused on its one job makes the codebase easy to test and maintain.

## Routing

### `routes/web.php` — Web routes (HTML responses, CSRF protected)

```php
use Espresso\Http\Controllers\PostController;
use League\Route\Router;

/** @var Router $router */

$router->map("GET",  "/posts",           [PostController::class, "index"]);
$router->map("GET",  "/posts/create",    [PostController::class, "create"]);
$router->map("POST", "/posts",           [PostController::class, "store"]);
$router->map("GET",  "/posts/{id}",      [PostController::class, "show"]);
$router->map("GET",  "/posts/{id}/edit", [PostController::class, "edit"]);
$router->map("POST", "/posts/{id}",      [PostController::class, "update"]);
$router->map("POST", "/posts/{id}/delete", [PostController::class, "destroy"]);
```

### `routes/api.php` — API routes (JSON responses, no CSRF)

```php
use Espresso\Http\Controllers\Api\PostController;

/** @var Router $router */

$router->map("GET",    "/api/posts",     [PostController::class, "index"]);
$router->map("POST",   "/api/posts",     [PostController::class, "store"]);
$router->map("GET",    "/api/posts/{id}", [PostController::class, "show"]);
$router->map("PUT",    "/api/posts/{id}", [PostController::class, "update"]);
$router->map("DELETE", "/api/posts/{id}", [PostController::class, "destroy"]);
```

> Routes under `/api/` are automatically excluded from CSRF protection.

### Route groups with middleware

Group related routes and apply middleware to all of them at once:

```php
$router->group("/admin", function ($router) {
    $router->map("GET", "/dashboard", [DashboardController::class, "index"]);
    $router->map("GET", "/users",     [UserController::class, "index"]);
})->middleware(new AuthMiddleware($container->get(AuthManager::class)));
```

### `#[Route]` attribute (alternative syntax)

Instead of defining routes in `routes/web.php`, you can annotate controller methods directly:

```php
use Espresso\Http\Attribute\Route;

class PostController extends AbstractController {
    #[Route("GET", "/posts")]
    public function index(ServerRequestInterface $request): ResponseInterface { ... }

    #[Route("POST", "/posts")]
    public function store(ServerRequestInterface $request): ResponseInterface { ... }
}
```

Then register the controller for scanning in `web.php`:

```php
$router->scanControllers([PostController::class]);
```

### URL parameters

Route parameters are accessed via `$request->getAttribute("id")`:

```php
$router->map("GET", "/posts/{id}", [PostController::class, "show"]);

// In controller:
$id = (int) $request->getAttribute("id");
```

## Controllers

Controllers receive requests, delegate validation to FormRequests, and call services.

### Creating a controller

```bash
php espresso make:controller Post
```

This generates `src/Http/Controllers/PostController.php` with `PostService` already injected.

### Manual controller

```php
<?php

declare(strict_types=1);

namespace Espresso\Http\Controllers;

use Espresso\Http\Controller\AbstractController;
use Espresso\Http\Requests\StorePostRequest;
use Espresso\Services\PostService;
use Espresso\Validation\Validator;
use Latte\Engine;
use Psr\Http\Message\ResponseInterface;
use Psr\Http\Message\ServerRequestInterface;

class PostController extends AbstractController {
    public function __construct(
        Engine $latte,
        Validator $validator,
        private readonly PostService $postService,
    ) {
        parent::__construct($latte, $validator);
    }

    public function index(ServerRequestInterface $request): ResponseInterface {
        $posts = $this->postService->all();
        return $this->view("posts/index.latte", ["posts" => $posts]);
    }

    public function store(ServerRequestInterface $request): ResponseInterface {
        $data = $this->formRequest(StorePostRequest::class, $request)->validated();
        $this->postService->create($data);
        return $this->redirect("/posts");
    }

    public function show(ServerRequestInterface $request): ResponseInterface {
        $id = (int) $request->getAttribute("id");
        $post = $this->postService->findOrFail($id);
        return $this->view("posts/show.latte", ["post" => $post]);
    }
}
```

### Available response helpers

| Method | Description |
|---|---|
| `$this->view("template.latte", $data)` | Render a Latte template |
| `$this->json($data, 200)` | Return a JSON response |
| `$this->redirect("/url")` | Return a redirect response |
| `$this->formRequest(RequestClass::class, $request)` | Validate via FormRequest |

### Register in ContainerFactory

Every controller that has dependencies beyond `Engine` and `Validator` must be registered in `src/Container/ContainerFactory.php`:

```php
PostController::class => function (Container $c): PostController {
    return new PostController(
        $c->get(Engine::class),
        $c->get(Validator::class),
        $c->get(PostService::class),
    );
},
```

## Form Requests (Validation)

FormRequests centralise validation rules so they never leak into controllers or services.

### Creating a request

```bash
php espresso make:request StorePost
```

This generates `src/Http/Requests/StorePostRequest.php`.

### Defining rules

Rules use [Respect\Validation](https://respect-validation.readthedocs.io):

```php
<?php

declare(strict_types=1);

namespace Espresso\Http\Requests;

use Espresso\Http\FormRequest;
use Respect\Validation\Validator as v;

class StorePostRequest extends FormRequest {
    protected function rules(): array {
        return [
            "title"   => v::stringType()->notEmpty()->length(1, 255),
            "body"    => v::stringType()->notEmpty(),
            "status"  => v::in(["draft", "published"]),
        ];
    }
}
```

Only the fields declared in `rules()` are returned by `validated()` — any extra input is automatically stripped.

### Using in a controller

```php
$data = $this->formRequest(StorePostRequest::class, $request)->validated();

// Access individual fields:
$fr = $this->formRequest(StorePostRequest::class, $request);
$title = $fr->get("title");
$all   = $fr->validated();
```

A `ValidationException` is thrown automatically when validation fails. `ExceptionMiddleware` catches it and returns a `422` response.

### Common Respect\Validation rules

```php
v::stringType()->notEmpty()            // non-empty string
v::intType()->min(1)                   // positive integer
v::email()                             // valid email
v::url()                               // valid URL
v::date("Y-m-d")                       // date format
v::length(3, 100)                      // string length range
v::in(["a", "b", "c"])                 // value in list
v::optional(v::stringType())          // nullable field
v::number()->between(0, 100)           // numeric range
```

## Services

Services contain your application's business logic. They sit between controllers and repositories and are the only layer that should make decisions about *what* to do.

### Creating a service

```bash
php espresso make:service Post
```

This generates `src/Services/PostService.php` with `PostRepository` injected.

### Manual service

```php
<?php

declare(strict_types=1);

namespace Espresso\Services;

use Doctrine\ORM\EntityManager;
use Espresso\Database\Entities\Post;
use Espresso\Database\Repository\PostRepository;
use Espresso\Http\Exception\HttpException;

class PostService extends AbstractService {
    public function __construct(
        EntityManager $entityManager,
        private readonly PostRepository $postRepository,
    ) {
        parent::__construct($entityManager);
    }

    public function all(): array {
        return $this->postRepository->findAll();
    }

    public function findOrFail(int $id): Post {
        $post = $this->postRepository->find($id);

        if (!$post instanceof Post) {
            throw new HttpException(404, "Post not found.");
        }

        return $post;
    }

    public function create(array $data): Post {
        $post = new Post($data["title"], $data["body"]);
        $this->postRepository->save($post);
        return $post;
    }

    public function update(int $id, array $data): Post {
        $post = $this->findOrFail($id);
        $post->setTitle($data["title"]);
        $post->setBody($data["body"]);
        $this->postRepository->save($post);
        return $post;
    }

    public function delete(int $id): void {
        $post = $this->findOrFail($id);
        $this->postRepository->delete($post);
    }
}
```

### Register in ContainerFactory

```php
use Espresso\Database\Repository\PostRepository;
use Espresso\Services\PostService;

PostRepository::class => function (Container $c): PostRepository {
    return new PostRepository($c->get(EntityManager::class));
},

PostService::class => function (Container $c): PostService {
    return new PostService(
        $c->get(EntityManager::class),
        $c->get(PostRepository::class),
    );
},
```

## Repositories

Repositories are the only place in the framework that directly queries the database. Everything else gets data through them.

### Creating a repository

```bash
php espresso make:repository Post
```

This generates `src/Database/Repository/PostRepository.php`.

### Manual repository

```php
<?php

declare(strict_types=1);

namespace Espresso\Database\Repository;

use Espresso\Database\Entities\Post;

class PostRepository extends AbstractRepository {
    protected function getEntityClass(): string {
        return Post::class;
    }

    public function findPublished(): array {
        return $this->findBy(["status" => "published"]);
    }

    public function findLatest(int $limit = 10): array {
        return $this->repository
            ->createQueryBuilder("p")
            ->where("p.status = :status")
            ->setParameter("status", "published")
            ->orderBy("p.createdAt", "DESC")
            ->setMaxResults($limit)
            ->getQuery()
            ->getResult();
    }
}
```

### Available base methods (from `AbstractRepository`)

| Method | Description |
|---|---|
| `find(int\|string $id)` | Find by primary key, returns `null` if not found |
| `findOrFail(int\|string $id)` | Find by primary key, throws `RuntimeException` if not found |
| `findAll()` | Return all records |
| `findBy(array $criteria, ?array $orderBy, ?int $limit, ?int $offset)` | Find by criteria |
| `findOneBy(array $criteria)` | Find a single record by criteria |
| `save(object $entity, bool $flush = true)` | Persist and optionally flush |
| `delete(object $entity, bool $flush = true)` | Remove and optionally flush |
| `flush()` | Flush pending changes to the database |

### Batch operations without auto-flush

When inserting many records, skip the per-save flush and do a single round-trip at the end:

```php
foreach ($items as $item) {
    $this->postRepository->save(new Post($item["title"]), flush: false);
}
$this->postRepository->flush(); // single DB round-trip
```

## Entities

Entities are PHP classes that map to database tables. Doctrine reads the `#[ORM\...]` attributes on each property to know which column it maps to.

### Creating an entity

```bash
php espresso make:entity Post
```

This generates `src/Database/Entities/Post.php` with an `id` column already set up.

### Full entity example

```php
<?php

declare(strict_types=1);

namespace Espresso\Database\Entities;

use DateTimeImmutable;
use Doctrine\ORM\Mapping as ORM;
use Espresso\Database\Concerns\HasMagicProperties;
use JsonSerializable;

#[ORM\Entity]
#[ORM\HasLifecycleCallbacks]
#[ORM\Table(name: "posts")]
class Post implements JsonSerializable {
    use HasMagicProperties;

    #[ORM\Id]
    #[ORM\GeneratedValue]
    #[ORM\Column(type: "integer")]
    private ?int $id = null;

    #[ORM\Column(type: "string", length: 255)]
    private string $title;

    #[ORM\Column(type: "text")]
    private string $body;

    #[ORM\Column(type: "string", length: 20)]
    private string $status = "draft";

    #[ORM\Column(type: "datetime_immutable")]
    private DateTimeImmutable $createdAt;

    public function __construct(string $title, string $body) {
        $this->title = $title;
        $this->body = $body;
        $this->createdAt = new DateTimeImmutable();
        $this->syncOriginal();
    }

    public function getId(): ?int { return $this->id; }
    public function getTitle(): string { return $this->title; }
    public function setTitle(string $title): void { $this->title = $title; }
    public function getBody(): string { return $this->body; }
    public function setBody(string $body): void { $this->body = $body; }
    public function getStatus(): string { return $this->status; }
    public function setStatus(string $status): void { $this->status = $status; }
    public function getCreatedAt(): DateTimeImmutable { return $this->createdAt; }
}
```

### Common Doctrine column types

| Type | PHP type | SQL |
|---|---|---|
| `integer` | `int` | INT |
| `string` | `string` | VARCHAR |
| `text` | `string` | TEXT |
| `boolean` | `bool` | TINYINT(1) |
| `float` | `float` | DOUBLE |
| `datetime_immutable` | `DateTimeImmutable` | DATETIME |
| `date_immutable` | `DateTimeImmutable` | DATE |
| `json` | `array` | JSON |

### Relationships

**One-to-Many (Post has many Comments):**

```php
// In Post entity
#[ORM\OneToMany(mappedBy: "post", targetEntity: Comment::class, cascade: ["persist", "remove"])]
private Collection $comments;

// In Comment entity
#[ORM\ManyToOne(targetEntity: Post::class, inversedBy: "comments")]
#[ORM\JoinColumn(nullable: false)]
private Post $post;
```

**Many-to-Many (Post has many Tags):**

```php
#[ORM\ManyToMany(targetEntity: Tag::class)]
#[ORM\JoinTable(name: "post_tags")]
private Collection $tags;
```

### Entity Magic Properties

The `HasMagicProperties` trait (in `src/Database/Concerns/`) gives every entity a set of convenient behaviours inspired by Laravel's Eloquent models. Add it to any entity along with `implements JsonSerializable` and the `#[ORM\HasLifecycleCallbacks]` attribute.

#### Dynamic property access

Instead of calling getters and setters explicitly, you can use property syntax:

```php
// Read — calls getTitle() / isCompleted() automatically
echo $post->title;
echo $post->is_completed; // snake_case also works

// Write — calls setTitle() automatically
$post->title = "New title";

// Check presence
isset($post->title); // true or false
```

#### Mass assignment with `fill()`

Populate multiple fields from an array at once — useful after form validation:

```php
$post->fill([
    "title"  => "Hello world",
    "body"   => "This is the body.",
    "status" => "published",
]);
```

Only fields that have a matching `set*()` method are updated; all others are silently ignored.

#### Serialisation

Convert an entity to an array or JSON string without writing manual mapping code:

```php
$post->toArray();
// ["id" => 1, "title" => "Hello", "created_at" => "2026-04-28T12:00:00+00:00", ...]

$post->toJson();
// '{"id":1,"title":"Hello",...}'

// Entities also implement JsonSerializable, so they serialise automatically:
json_encode($post);
json_encode(["post" => $post]);
```

Dates are automatically formatted as ISO 8601 strings.

#### Dirty tracking

Know exactly what changed between the time an entity was loaded and the current moment:

```php
$post = $postRepository->find(1);
$post->setTitle("Updated title");

$post->isDirty();          // true  — something changed
$post->isDirty("title");   // true  — this specific property changed
$post->isDirty("body");    // false — body has not changed

$post->getOriginal("title");  // "Original title"
$post->getOriginal();         // full snapshot array as loaded from the DB
```

This is useful in services when you need to act only on properties that actually changed, for example when triggering notifications or writing audit logs.

## Migrations

Migrations are versioned SQL changes. Doctrine generates them automatically by comparing your entities against the current database schema.

### Workflow

```bash
# 1. Create or modify an entity

# 2. Generate a migration from the entity diff
php espresso migrations:diff

# 3. Review the generated file in database/migrations/

# 4. Apply the migration
php espresso migrations:migrate

# 5. Check status at any time
php espresso migrations:status
```

### Create a blank migration manually

```bash
php espresso make:migration CreatePostsTable
```

### Migration file structure

```php
<?php

declare(strict_types=1);

namespace Database\Migrations;

use Doctrine\DBAL\Schema\Schema;
use Doctrine\Migrations\AbstractMigration;

final class Version20260428120000 extends AbstractMigration {
    public function getDescription(): string {
        return "Create posts table";
    }

    public function up(Schema $schema): void {
        $this->addSql("CREATE TABLE posts (
            id INT AUTO_INCREMENT NOT NULL,
            title VARCHAR(255) NOT NULL,
            body LONGTEXT NOT NULL,
            status VARCHAR(20) NOT NULL DEFAULT 'draft',
            created_at DATETIME NOT NULL COMMENT '(DC2Type:datetime_immutable)',
            PRIMARY KEY (id)
        ) DEFAULT CHARACTER SET utf8mb4 ENGINE = InnoDB");
    }

    public function down(Schema $schema): void {
        $this->addSql("DROP TABLE posts");
    }
}
```

### Run a specific version

```bash
php espresso migrations:execute --up 'Database\Migrations\Version20260428120000'
php espresso migrations:execute --down 'Database\Migrations\Version20260428120000'
```

## Seeders

Seeders populate the database with test or default data.

### Create a seeder

Create `database/seeders/PostSeeder.php`:

```php
<?php

declare(strict_types=1);

namespace Database\Seeders;

use Doctrine\ORM\EntityManager;
use Espresso\Database\Entities\Post;

class PostSeeder {
    public function __construct(private readonly EntityManager $entityManager) {}

    public function run(): void {
        $posts = [
            ["title" => "First Post", "body" => "Hello world."],
            ["title" => "Second Post", "body" => "Another post."],
        ];

        foreach ($posts as $data) {
            $post = new Post($data["title"], $data["body"]);
            $this->entityManager->persist($post);
        }

        $this->entityManager->flush();
    }
}
```

### Create a seed console command

Create `src/Console/Commands/SeedCommand.php`:

```php
<?php

declare(strict_types=1);

namespace Espresso\Console\Commands;

use Database\Seeders\PostSeeder;
use Doctrine\ORM\EntityManager;
use Symfony\Component\Console\Command\Command;
use Symfony\Component\Console\Input\InputInterface;
use Symfony\Component\Console\Output\OutputInterface;

class SeedCommand extends Command {
    public function __construct(private readonly EntityManager $entityManager) {
        parent::__construct();
    }

    protected function configure(): void {
        $this->setName("db:seed")->setDescription("Run database seeders");
    }

    protected function execute(InputInterface $input, OutputInterface $output): int {
        (new PostSeeder($this->entityManager))->run();
        $output->writeln("<info>Database seeded.</info>");
        return Command::SUCCESS;
    }
}
```

Then register it in `ContainerFactory` and add it to `Console\Kernel`.

## Frontend Assets (Vite + npm)

Espresso uses [Vite](https://vitejs.dev) to bundle CSS and JavaScript. Source files live in `resources/` and compiled output goes to `public/build/`.

### First-time setup

```bash
npm install
```

### Development

```bash
npm run dev
```

Vite starts a hot-reloading asset server at `http://localhost:5173`. The `vite()` helper in your layout automatically points to this server during development so changes appear instantly without a page refresh.

### Production build

```bash
npm run build
```

Compiled, fingerprinted files are written to `public/build/`. The `vite()` helper switches to these files automatically when the manifest is present.

### Adding CSS

All styles start in `resources/css/app.css`. Import additional stylesheets there:

```css
@import "./buttons.css";
@import "./forms.css";
```

### Adding JavaScript

All scripts start in `resources/js/app.js`. Import additional modules there:

```js
import "./bootstrap";
import Alpine from "alpinejs";

Alpine.start();
```

Install npm packages the usual way:

```bash
npm install alpinejs
```

### Including assets in a template

The `vite()` helper is already called in `resources/views/layouts/app.latte`, so you only need to add it manually if you create a layout that does not extend the default one:

```latte
{vite("resources/js/app.js")|noescape}
```

In development this outputs the Vite dev server script tags. After a production build it outputs the hashed `<link>` and `<script>` tags from the manifest.

## Views (Latte)

Templates live in `resources/views/`. Espresso uses [Latte](https://latte.nette.org) as its template engine.

### Extending the base layout

Create a new template that inherits the nav, footer, and asset loading from `layouts/app.latte`:

```latte
{extends "layouts/app.latte"}

{block title}Posts{/block}

{block content}
<h1>All Posts</h1>

{foreach $posts as $post}
  <div>{$post->title}</div>
{/foreach}
{/block}
```

### Latte syntax quick reference

| Task | Syntax |
|---|---|
| Print a variable | `{$variable}` |
| Print HTML (unescaped) | `{$html\|noescape}` |
| Call a function | `{=csrf_field()}` |
| Conditional | `{if $condition}...{else}...{/if}` |
| Loop | `{foreach $items as $item}...{/foreach}` |
| Extend layout | `{extends "layouts/app.latte"}` |
| Define a block | `{block name}...{/block}` |
| Comment | `{* this is a comment *}` |

### Available global variables

| Variable | Description |
|---|---|
| `{$app_name}` | Value of `APP_NAME` |
| `{$app_env}` | Value of `APP_ENV` |
| `{$auth}` | The `AuthManager` instance |
| `{$auth->check($request)}` | `true` if a user is authenticated |

### CSRF protection

Include a hidden CSRF field inside every HTML form that submits data:

```latte
<form action="/posts" method="POST">
  {=csrf_field()}

  <input type="text" name="title">
  <button type="submit">Save</button>
</form>
```

For AJAX requests, read the raw token:

```latte
<meta name="csrf-token" content="{=csrf_token()}">
```

```js
fetch("/api/posts", {
    method: "POST",
    headers: { "X-CSRF-Token": document.querySelector('meta[name="csrf-token"]').content },
});
```

### Passing data from a controller

```php
return $this->view("posts/show.latte", [
    "post"     => $post,
    "comments" => $comments,
]);
```

```latte
<h1>{$post->title}</h1>
<p>{$post->body}</p>

{foreach $comments as $comment}
  <div>{$comment->body}</div>
{/foreach}
```

> **Tip:** With `HasMagicProperties` on your entities, you can use `$post->title` in templates instead of `$post->getTitle()`, making templates much cleaner.

### Page-specific styles and scripts

Use the `styles` and `scripts` blocks to add page-specific CSS or JS without polluting the global layout:

```latte
{block styles}
<style>
  .hero { background: #1e293b; color: white; padding: 4rem 0; }
</style>
{/block}

{block scripts}
<script>
  console.log("Page loaded");
</script>
{/block}
```

## Middleware

Middleware runs before and/or after every request. Use it for cross-cutting concerns like authentication, logging, and CORS.

### Creating middleware

```php
<?php

declare(strict_types=1);

namespace Espresso\Http\Middleware;

use Psr\Http\Message\ResponseInterface;
use Psr\Http\Message\ServerRequestInterface;
use Psr\Http\Server\MiddlewareInterface;
use Psr\Http\Server\RequestHandlerInterface;

class LogRequestMiddleware implements MiddlewareInterface {
    public function process(ServerRequestInterface $request, RequestHandlerInterface $handler): ResponseInterface {
        // Before request
        $response = $handler->handle($request);
        
        // After response
        return $response;
    }
}
```

### Attaching middleware to a single route

```php
$router->map("GET", "/dashboard", [DashboardController::class, "index"])
    ->middleware(new AuthMiddleware($container->get(AuthManager::class)));
```

### Attaching middleware to a group

```php
$router->group("/admin", function ($router) {
    $router->map("GET", "/users",    [UserController::class, "index"]);
    $router->map("GET", "/settings", [SettingsController::class, "index"]);
})->middleware(new AuthMiddleware($container->get(AuthManager::class), "web"));
```

## Authentication

### Session-based (web)

```php
$authManager = $container->get(AuthManager::class);
$guard = $authManager->guard("web");

// Login
$guard->login($userEntity, $request);

// Attempt with credentials
$success = $guard->attempt("user@example.com", "password", UserEntity::class, $request);

// Check if authenticated
$guard->check($request); // bool

// Get authenticated user
$user = $guard->user($request); // returns Authenticatable or null

// Logout
$guard->logout($request);
```

### JWT (API)

```php
$guard = $authManager->guard("api");

// Generate token (returns JWT string)
$token = $guard->attempt("user@example.com", "password", UserEntity::class, $request);

// Validate token from Authorization header
$guard->check($request); // bool
$user = $guard->user($request);
```

### Protecting routes with AuthMiddleware

```php
use Espresso\Http\Middleware\AuthMiddleware;
use Espresso\Auth\AuthManager;

// web guard (session)
$router->map("GET", "/profile", [ProfileController::class, "index"])
    ->middleware(new AuthMiddleware($container->get(AuthManager::class), "web"));

// api guard (JWT)
$router->map("GET", "/api/profile", [Api\ProfileController::class, "index"])
    ->middleware(new AuthMiddleware($container->get(AuthManager::class), "api"));
```

### Making an entity authenticatable

```php
use Espresso\Auth\Contracts\Authenticatable;

#[ORM\Entity]
#[ORM\Table(name: "users")]
class User implements Authenticatable {
    // ... columns ...

    public function getId(): int|string { return $this->id; }
    public function getEmail(): string { return $this->email; }
    public function getPassword(): string { return $this->password; }
}
```

## Configuration Reference

All config files live in `config/` and are loaded automatically on boot.

---

### App (`config/app.php`)

```php
return [
    "name"    => $_ENV["APP_NAME"] ?? "Framework",
    "env"     => $_ENV["APP_ENV"] ?? "production",  // local | production | testing
    "debug"   => filter_var($_ENV["APP_DEBUG"] ?? false, FILTER_VALIDATE_BOOLEAN),
    "url"     => $_ENV["APP_URL"] ?? "http://localhost",
    "key"     => $_ENV["APP_KEY"] ?? "",
    "views"   => dirname(__DIR__) . "/resources/views",
    "storage" => dirname(__DIR__) . "/storage",
];
```

---

### Database (`config/database.php`)

Switch the active driver with `DB_CONNECTION` in `.env`.

**MySQL:**

```env
DB_CONNECTION=mysql
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=my_database
DB_USERNAME=root
DB_PASSWORD=secret
```

**SQLite:**

```env
DB_CONNECTION=sqlite
DB_DATABASE=/absolute/path/to/database.sqlite
# or relative (leave empty to use database/database.sqlite):
DB_DATABASE=
```

**PostgreSQL:**

```env
DB_CONNECTION=pgsql
DB_HOST=127.0.0.1
DB_PORT=5432
DB_DATABASE=my_database
DB_USERNAME=postgres
DB_PASSWORD=secret
```

**Adding a custom driver:**

1. Add a connection entry in `config/database.php`:

```php
"connections" => [
    // ...existing...
    "sqlsrv" => [
        "driver"   => "pdo_sqlsrv",
        "host"     => $_ENV["DB_HOST"] ?? "127.0.0.1",
        "port"     => (int) ($_ENV["DB_PORT"] ?? 1433),
        "dbname"   => $_ENV["DB_DATABASE"] ?? "framework",
        "user"     => $_ENV["DB_USERNAME"] ?? "sa",
        "password" => $_ENV["DB_PASSWORD"] ?? "",
    ],
],
```

2. Set `DB_CONNECTION=sqlsrv` in `.env`.

### Cache (`config/cache.php`)

Switch the active driver with `CACHE_DRIVER` in `.env`.

**File (default):**

```env
CACHE_DRIVER=file
```

**Redis:**

```env
CACHE_DRIVER=redis
REDIS_HOST=127.0.0.1
REDIS_PORT=6379
REDIS_PASSWORD=
```

**APCu:**

```env
CACHE_DRIVER=apcu
```

**Using the cache in a service:**

```php
use Symfony\Contracts\Cache\CacheInterface;
use Symfony\Contracts\Cache\ItemInterface;

class PostService extends AbstractService {
    public function __construct(
        EntityManager $entityManager,
        private readonly PostRepository $postRepository,
        private readonly CacheInterface $cache,
    ) {
        parent::__construct($entityManager);
    }

    public function all(): array {
        return $this->cache->get("posts.all", function (ItemInterface $item): array {
            $item->expiresAfter(3600);
            return $this->postRepository->findAll();
        });
    }
}
```

Register `CacheInterface` in the constructor when binding in `ContainerFactory`.

---

### Logging (`config/logging.php`)

Switch the active channel with `LOG_CHANNEL` in `.env`.

**File log:**

```env
LOG_CHANNEL=file
LOG_LEVEL=debug   # debug | info | notice | warning | error | critical
```

Log file is written to `storage/logs/app.log`.

**PHP error_log:**

```env
LOG_CHANNEL=errorlog
LOG_LEVEL=error
```

**Using the logger:**

```php
use Psr\Log\LoggerInterface;

class PostService extends AbstractService {
    public function __construct(
        EntityManager $entityManager,
        private readonly PostRepository $postRepository,
        private readonly LoggerInterface $logger,
    ) {
        parent::__construct($entityManager);
    }

    public function create(array $data): Post {
        $post = new Post($data["title"], $data["body"]);
        $this->postRepository->save($post);
        $this->logger->info("Post created", ["id" => $post->getId()]);
        return $post;
    }
}
```

### Authentication (`config/auth.php`)

**Session guard (web):**

```env
AUTH_GUARD=web
SESSION_LIFETIME=7200
SESSION_COOKIE=framework_session
```

**JWT guard (API):**

```env
AUTH_GUARD=api

```

**Switching guards:** Call `$authManager->guard("api")` or `$authManager->guard("web")` explicitly at any time regardless of the default.

## Container Bindings

All services are resolved from the PHP-DI container. When you create a new class with dependencies, register it in `src/Container/ContainerFactory.php` inside `definitions()`:

```php
use Espresso\Services\PostService;
use Espresso\Database\Repository\PostRepository;
use Espresso\Http\Controllers\PostController;

PostRepository::class => function (Container $c): PostRepository {
    return new PostRepository($c->get(EntityManager::class));
},

PostService::class => function (Container $c): PostService {
    return new PostService(
        $c->get(EntityManager::class),
        $c->get(PostRepository::class),
    );
},

PostController::class => function (Container $c): PostController {
    return new PostController(
        $c->get(Engine::class),
        $c->get(Validator::class),
        $c->get(PostService::class),
    );
},
```

Resolving from the container manually (e.g. in a seeder command):

```php
$postService = $container->get(PostService::class);
```

The standard registration order is: **Repository → Service → Controller**, since each depends on the one before it.
